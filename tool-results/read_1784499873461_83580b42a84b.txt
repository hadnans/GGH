    29вҶ’## 1. Database Philosophy
    30вҶ’
    31вҶ’| Principle | Rule | Why |
    32вҶ’|---|---|---|
    33вҶ’| **Medusa owns its schema** | The `public` schema is managed entirely by Medusa's migration engine. GGH never modifies Medusa tables directly. | Medusa upgrades will break if we alter their tables. Custom data lives in our own schema. |
    34вҶ’| **GGH data is separate** | All GGH-specific tables live in the `ggh` schema. Cross-schema references use foreign keys to Medusa's IDs stored as text. | Clean separation. GGH migrations don't conflict with Medusa migrations. |
    35вҶ’| **Integers for money** | All monetary values are stored as piastres (1/100 of a pound) in `INTEGER` columns. No `DECIMAL`, no `FLOAT`, no `NUMERIC`. | Floating-point arithmetic causes rounding errors. EGP 110.50 becomes `11050` вҖ” exact, always. |
    36вҶ’| **UUIDs for primary keys** | Every table uses `UUID` primary keys generated with `gen_random_uuid()`. No auto-incrementing integers. | UUIDs are globally unique, safe to generate client-side, and don't leak record counts. |
    37вҶ’| **Soft delete by default** | Records are never physically deleted. A `deleted_at` timestamp marks deletion. | Orders, products, and customers must be recoverable. Physical deletion breaks referential integrity. |
    38вҶ’| **Timestamps on everything** | Every table has `created_at` and `updated_at`. Every sync table has a `created_at` only (immutable logs). | Audit trail. Debugging. Replication. No exceptions. |
    39вҶ’| **Bilingual fields are paired** | Every user-facing text field has `_en` and `_ar` companions. | Product names, category names, and zone names exist in both languages. Never store "current locale" вҖ” store both. |
    40вҶ’| **No arrays in columns** | No `ARRAY` column types. Use junction tables for many-to-many relationships. | Arrays in columns are not queryable, not indexable, and not relational. Junction tables are. |

    44вҶ’## 2. Naming Conventions
    45вҶ’
    46вҶ’### 2.1 Table Naming
    47вҶ’
    48вҶ’| Entity Type | Convention | Example |
    49вҶ’|---|---|---|
    50вҶ’| Medusa core tables | Managed by Medusa (snake_case) | `product`, `product_variant`, `order`, `customer` |
    51вҶ’| GGH custom tables | `snake_case`, prefixed by domain | `delivery_zone`, `delivery_slot`, `price_tier` |
    52вҶ’| Junction tables | Both entity names, alphabetically | `product_category`, `customer_group_customer` |
    53вҶ’| Audit/log tables | Entity name + `_log` | `erp_sync_log`, `order_status_log` |
    54вҶ’| Schema name | `ggh` | `ggh.delivery_zone`, `ggh.erp_sync_log` |
    55вҶ’
    56вҶ’### 2.2 Column Naming
    57вҶ’
    58вҶ’| Pattern | Convention | Example |
    59вҶ’|---|---|---|
    60вҶ’| Primary key | `id` | `id` (always UUID) |
    61вҶ’| Foreign key | Referenced table singular + `_id` | `customer_id`, `product_id`, `zone_id` |
    62вҶ’| Timestamp | `_at` suffix | `created_at`, `updated_at`, `deleted_at`, `delivered_at` |
    63вҶ’| Boolean | `is_` or `has_` prefix | `is_active`, `is_case_pack`, `has_free_delivery` |
    64вҶ’| Money amount | `_amount` suffix, integer (piastres) | `base_rate_amount`, `total_amount`, `savings_amount` |
    65вҶ’| Bilingual text | `_en` and `_ar` suffix pair | `name_en` / `name_ar`, `description_en` / `description_ar` |
    66вҶ’| JSON data | `_data` suffix | `regions_data`, `items_data`, `metadata_data` |
    67вҶ’| Count | `_count` suffix | `max_orders_count`, `current_orders_count` |
    68вҶ’| Enumeration | Column name only (string with check constraint) | `status`, `direction`, `entity_type` |
    69вҶ’
    70вҶ’### 2.3 Constraint Naming
    71вҶ’
    72вҶ’| Constraint Type | Convention | Example |
    73вҶ’|---|---|---|
    74вҶ’| Primary key | `pk_<table>` | `pk_delivery_zone` |
    75вҶ’| Foreign key | `fk_<table>_<referenced_table>` | `fk_delivery_slot_zone` |
    76вҶ’| Unique | `uq_<table>_<columns>` | `uq_price_tier_variant_min_qty` |
    77вҶ’| Check | `ck_<table>_<column>` | `ck_delivery_slot_max_orders` |
    78вҶ’| Index | `ix_<table>_<columns>` | `ix_erp_sync_log_entity_status` |

   182вҶ’## 5. Core Commerce Entities
   183вҶ’
   184вҶ’These entities are owned by Medusa. GGH reads and writes them through the Medusa API, never through direct database access. The definitions below describe how GGH uses them, not their full Medusa schema.
   185вҶ’
   186вҶ’### 5.1 Product
   187вҶ’
   188вҶ’| Aspect | GGH Usage |
   189вҶ’|---|---|
   190вҶ’| **Purpose** | Canonical product definition (e.g., "Al Doha Basmati Rice") |
   191вҶ’| **Key fields used** | `id`, `title`, `handle`, `description`, `thumbnail`, `status`, `category_id`, `metadata` |
   192вҶ’| **GGH metadata** | `metadata.name_ar` вҖ” Arabic product name (not natively supported by Medusa) |
   193вҶ’| **GGH metadata** | `metadata.brand` вҖ” Brand name (e.g., "Al Doha", "El Maleka") |
   194вҶ’| **GGH metadata** | `metadata.erp_item_code` вҖ” ERPNext Item Code for sync |
   195вҶ’| **GGH metadata** | `metadata.moq` вҖ” Minimum order quantity for this product |
   196вҶ’| **Created by** | ERP sync (primary) or Medusa Admin (manual) |
   197вҶ’| **Soft deleted** | Yes вҖ” Medusa sets `deleted_at` |
   198вҶ’
   199вҶ’### 5.2 Product Variant
   200вҶ’
   201вҶ’| Aspect | GGH Usage |
   202вҶ’|---|---|
   203вҶ’| **Purpose** | A specific pack size of a product (e.g., "1kg", "5kg", "25kg") |
   204вҶ’| **Key fields used** | `id`, `product_id`, `title`, `sku`, `barcode`, `allow_backorder`, `metadata` |
   205вҶ’| **GGH metadata** | `metadata.weight_kg` вҖ” Weight in kilograms |
   206вҶ’| **GGH metadata** | `metadata.unit` вҖ” Unit of measure ("kg", "L", "pack") |
   207вҶ’| **GGH metadata** | `metadata.is_case_pack` вҖ” Whether this is a case-pack variant |
   208вҶ’| **GGH metadata** | `metadata.case_size` вҖ” Number of units in a case (if case pack) |
   209вҶ’| **GGH metadata** | `metadata.moq` вҖ” Variant-level minimum order quantity |
   210вҶ’| **Relationship** | Belongs to Product. Has many PriceTiers (ggh). Has one InventoryLevel. |
   211вҶ’
   212вҶ’### 5.3 Product Category
   213вҶ’
   214вҶ’| Aspect | GGH Usage |
   215вҶ’|---|---|
   216вҶ’| **Purpose** | Product grouping (e.g., "Rice & Grains", "Oils & Fats") |
   217вҶ’| **Key fields used** | `id`, `name`, `handle`, `parent_category_id`, `rank`, `metadata` |
   218вҶ’| **GGH metadata** | `metadata.name_ar` вҖ” Arabic category name |
   219вҶ’| **GGH metadata** | `metadata.icon_emoji` вҖ” Emoji for category display (рҹҚҡ, рҹ«’, вҳ•) |
   220вҶ’| **GGH metadata** | `metadata.erp_item_group` вҖ” ERPNext Item Group name for sync |
   221вҶ’| **Structure** | Two-level hierarchy only: parent category вҶ’ child categories. No deeper nesting. |

   784вҶ’## 10. Index Strategy
   785вҶ’
   786вҶ’### 10.1 Index Principles
   787вҶ’
   788вҶ’| Principle | Detail |
   789вҶ’|---|---|
   790вҶ’| **Index for query patterns, not structure** | Add an index when a query is slow, not because a column "might be searched." |
   791вҶ’| **Prefer composite indexes** | A single composite index covering multiple WHERE clauses beats multiple single-column indexes. |
   792вҶ’| **Include only what's needed** | Use covering indexes (INCLUDE columns) when the query only needs a few columns. |
   793вҶ’| **Monitor and remove unused indexes** | Every index slows writes. Unused indexes are waste. Check `pg_stat_user_indexes` monthly. |
   794вҶ’| **No indexes on small tables** | Tables under 1000 rows don't need indexes. Sequential scan is faster. |

   904вҶ’## 12. Soft Delete Strategy
   905вҶ’
   906вҶ’### 12.1 Rules
   907вҶ’
   908вҶ’| Rule | Detail |
   909вҶ’|---|---|
   910вҶ’| **All GGH tables use soft delete** | Every table has `deleted_at TIMESTAMPTZ NULL`. Deleted records have a timestamp; active records have NULL. |
   911вҶ’| **No physical deletion** | Records are never removed with DELETE. Only UPDATE SET deleted_at = now(). |
   912вҶ’| **Unique constraints exclude soft-deleted** | All unique constraints use `WHERE deleted_at IS NULL` to allow re-creation after soft delete. |
   913вҶ’| **Queries filter by default** | All application queries include `WHERE deleted_at IS NULL`. Use a base scope or middleware. |
   914вҶ’| **Hard delete after 90 days** | A scheduled job permanently removes records soft-deleted more than 90 days ago. |
   915вҶ’| **Log tables have no soft delete** | `erp_sync_log` and `order_status_log` are append-only. They are never soft- or hard-deleted. |
   916вҶ’
   917вҶ’### 12.2 Soft Delete Cascade
   918вҶ’
   919вҶ’When a parent record is soft-deleted, related records are also soft-deleted:
   920вҶ’
   921вҶ’| Parent | Children Soft-Deleted |
   922вҶ’|---|---|
   923вҶ’| delivery_zone | All delivery_slots in this zone |

   930вҶ’## 13. Money Handling
   931вҶ’
   932вҶ’### 13.1 Storage Rules
   933вҶ’
   934вҶ’| Rule | Detail |
   935вҶ’|---|---|
   936вҶ’| **All money is stored in piastres** | EGP 125.50 вҶ’ `12550`. EGP 0.00 вҶ’ `0`. |
   937вҶ’| **Column type is INTEGER** | Not DECIMAL, not NUMERIC, not FLOAT. Integer arithmetic is exact. |
   938вҶ’| **No negative amounts** | All amount columns have a CHECK constraint вүҘ 0. Refunds are tracked separately, not as negative amounts. |
   939вҶ’| **Currency is always EGP** | The `currency_code` column exists for future-proofing but is always "EGP" in practice. |
   940вҶ’| **Display conversion happens at render time** | `formatEGP(12550)` вҶ’ "125.50 Ш¬.Щ…". The database never stores formatted strings. |

  1005вҶ’## 15. Audit & Timestamps
  1006вҶ’
  1007вҶ’### 15.1 Timestamp Rules
  1008вҶ’
  1009вҶ’| Table Type | Timestamps | Mutable? |
  1010вҶ’|---|---|---|
  1011вҶ’| Entity tables (delivery_zone, etc.) | `created_at`, `updated_at`, `deleted_at` | `updated_at` updated on every change. `deleted_at` set on soft delete. |
  1012вҶ’| Log tables (erp_sync_log, order_status_log) | `created_at` only | Immutable. Never updated or deleted. |
  1013вҶ’| Cursor table (erp_sync_cursor) | `updated_at` | Updated after every successful sync. |
  1014вҶ’
  1015вҶ’### 15.2 Audit Trail
  1016вҶ’
  1017вҶ’| Event | Audit Source |
  1018вҶ’|---|---|
  1019вҶ’| Product created/updated/deleted | Medusa's built-in audit log |
  1020вҶ’| Order status changed | `ggh.order_status_log` |
  1021вҶ’| ERP sync attempt | `ggh.erp_sync_log` |
  1022вҶ’| Customer verified | `ggh.wholesale_account.verified_by` + `verified_at` |
  1023вҶ’| Delivery slot count changed | Application-level logging (slot increment is atomic) |
  1024вҶ’| Price tier changed | `ggh.price_tier.updated_at` + application-level change log |
  1025вҶ’
  1026вҶ’### 15.3 No Triggers
  1027вҶ’
  1028вҶ’GGH does not use database triggers. All audit logging and derived updates happen in the application layer. Reasons:
  1029вҶ’
  1030вҶ’1. Triggers are invisible вҖ” a developer reading the code cannot see the side effect
  1031вҶ’2. Triggers are hard to test вҖ” they require database-level integration tests
  1032вҶ’3. Triggers are hard to debug вҖ” they don't appear in application logs
  1033вҶ’4. Triggers create coupling вҖ” the database knows about business rules

  1037вҶ’## 16. Migration Strategy
  1038вҶ’
  1039вҶ’### 16.1 Medusa Migrations
  1040вҶ’
  1041вҶ’| Rule | Detail |
  1042вҶ’|---|---|
  1043вҶ’| **Never modify Medusa migrations** | Run `medusa db:migrate` to apply. Never edit existing migration files. |
  1044вҶ’| **Custom Medusa migrations** | If GGH needs to add a column to a Medusa table, create a new migration file in Medusa's migration directory. |
  1045вҶ’| **Test on staging first** | Run all migrations on a staging database before production. |
  1046вҶ’| **Backup before migration** | Take a PostgreSQL dump before running migrations in production. |
  1047вҶ’
  1048вҶ’### 16.2 GGH Migrations
  1049вҶ’
  1050вҶ’| Rule | Detail |
  1051вҶ’|---|---|
  1052вҶ’| **Numbered migration files** | `001_create_delivery_zone.sql`, `002_create_delivery_slot.sql`, etc. |
  1053вҶ’| **Each migration is reversible** | Every migration has an `UP` (create) and `DOWN` (revert) section. |
  1054вҶ’| **Migrations are idempotent** | Check if the table/column already exists before creating. |
  1055вҶ’| **No data modification in migrations** | Schema changes only. Data seeding uses `scripts/seed/`. |
  1056вҶ’| **Run in order** | Migration runner executes files in numerical order. Never skip a number. |
  1057вҶ’
  1058вҶ’### 16.3 Migration File Template
  1059вҶ’
  1060вҶ’```
  1061вҶ’-- Migration: 001_create_delivery_zone
  1062вҶ’-- Created: 2026-07-19
  1063вҶ’-- Author: GGH Team
  1064вҶ’-- Description: Create delivery_zone table for shipping zone management
  1065вҶ’
  1066вҶ’-- UP
  1067вҶ’CREATE TABLE IF NOT EXISTS ggh.delivery_zone ( ... );
  1068вҶ’CREATE INDEX IF NOT EXISTS ... ;
  1069вҶ’
  1070вҶ’-- DOWN
  1071вҶ’DROP INDEX IF EXISTS ... ;
  1072вҶ’DROP TABLE IF EXISTS ggh.delivery_zone;
  1073вҶ’```

  1078вҶ’
  1079вҶ’### 17.1 Backup Schedule
  1080вҶ’
  1081вҶ’| Type | Frequency | Retention | Tool |
  1082вҶ’|---|---|---|---|
  1083вҶ’| Full dump | Daily at 03:00 UTC | 30 days | `pg_dump` |
  1084вҶ’| WAL archiving | Continuous | 7 days | PostgreSQL WAL |
  1085вҶ’| Snapshot (if cloud) | Daily at 04:00 UTC | 14 days | Cloud provider snapshot |
  1086вҶ’
  1087вҶ’### 17.2 Recovery Time Objectives
  1088вҶ’
  1089вҶ’| Scenario | RTO | RPO | Method |
  1090вҶ’|---|---|---|---|
  1091вҶ’| Single table drop | 15 minutes | 0 (WAL replay) | Point-in-time recovery |
  1092вҶ’| Full database loss | 1 hour | вүӨ 24 hours | Restore from latest dump + WAL replay |
  1093вҶ’| Data corruption (detected within 1h) | 30 minutes | вүӨ 1 hour | Point-in-time recovery to before corruption |
  1094вҶ’| Data corruption (detected after 24h) | 4 hours | вүӨ 24 hours | Restore from dump + manual reconciliation |
  1095вҶ’
  1096вҶ’### 17.3 Backup Verification
  1097вҶ’
  1098вҶ’| Check | Frequency | Method |
  1099вҶ’|---|---|---|
  1100вҶ’| Backup file exists and is non-empty | Every run | Script checks file size > 0 |
  1101вҶ’| Backup is restorable | Weekly | Restore to a test database, run integrity checks |
  1102вҶ’| GGH schema is present | Weekly | Verify `ggh.delivery_zone` exists in restored backup |
  1103вҶ’| Row counts match | Weekly | Compare row counts on key tables between live and restored |

    35→## 1. Structure Philosophy
    36→
    37→Before listing folders, the principles that govern where files go:
    38→
    39→| Principle | Rule | Why |
    40→|---|---|---|
    41→| **Feature locality** | Files that change together live together. A product card component, its hooks, and its types belong in the same feature directory, not scattered across three top-level folders. | Reduces cognitive load. When you fix a bug in "product card," you open one folder, not five. |
    42→| **Dependency direction** | Dependencies flow inward. `features/` depends on `components/` and `lib/`. `components/` depends on `lib/` and `types/`. `lib/` depends on nothing internal. `utils/` depends on nothing at all. | Prevents circular dependencies. Makes the dependency graph a tree, not a web. |
    43→| **No orphan files** | Every file has a home. If you cannot decide where a file belongs, the folder structure is wrong — fix the structure, not the file. | Prevents `misc/`, `shared/`, and `common/` from becoming junk drawers. |
    44→| **Elder-friendly for developers too** | A new developer should find any file within 10 seconds by following the folder naming convention. No hidden files. No clever aliases. | If Om Ibrahim needs clear navigation, so does the junior developer joining the team. |
    45→| **RTL-first organization** | Localization files are first-class citizens, not an afterthought in a `lang/` folder at the bottom. The `locales/` directory sits at the same level as `components/` and `features/`. | Arabic is the primary language. The folder structure must reflect this priority. |
    46→
    47→### Dependency Graph
    48→
    49→```
    50→utils/          ← Pure functions. Zero dependencies on anything internal.
    51→  │
    52→  ▼
    53→types/          ← Type definitions. May import from utils/ for branded types.
    54→  │
    55→  ▼
    56→lib/            ← Core libraries, clients, configurations. Imports types/, utils/.
    57→  │
    58→  ▼
    59→styles/         ← Design tokens, global CSS. No imports from features/ or components/.
    60→  │
    61→  ▼
    62→locales/        ← Translation files. No code imports. Consumed by next-intl at build time.
    63→  │
    64→  ▼
    65→components/     ← Reusable UI primitives. Imports lib/, types/, utils/. Never imports features/.
    66→  │
    67→  ▼
    68→services/       ← API call functions. Imports lib/, types/. Never imports components/ or features/.
    69→  │
    70→  ▼
    71→hooks/          ← Shared React hooks. Imports lib/, services/, types/. Never imports features/.
    72→  │
    73→  ▼
    74→providers/      ← Context wrappers. Imports hooks/, lib/, types/.
    75→  │
    76→  ▼
    77→features/       ← Domain modules. Imports everything above. Nothing imports from features/.
    78→  │
    79→  ▼
    80→app/            ← Next.js routing. Imports features/, components/, providers/. The top of the tree.
    81→```

   147→## 4. frontend/src/app/ — Routing & Pages
   148→
   149→### Purpose
   150→
   151→The `app/` directory is the Next.js App Router. It defines every URL the user can visit, every API endpoint the frontend exposes, and the layout hierarchy that wraps them. This folder maps directly to the browser's address bar.
   152→
   153→### What Belongs Here
   154→
   155→| Subdirectory | Contents |
   156→|---|---|
   157→| Route groups `(shop)/`, `(auth)/`, `(account)/` | Page components, layouts, loading states, error boundaries |
   158→| `api/` | Route handlers (BFF endpoints) |
   159→| `layout.tsx` | Root layout with providers, fonts, and HTML attributes (`dir`, `lang`) |
   160→| `globals.css` | Tailwind directives and CSS custom properties |
   161→| `not-found.tsx` | Custom 404 page |
   162→| `error.tsx` | Global error boundary |
   163→| `loading.tsx` | Global loading skeleton |
   164→
   165→### What Should NOT Be Here
   166→
   167→| Anti-pattern | Why |
   168→|---|---|
   169→| Business logic | Pages import from `features/` and delegate. They do not contain logic themselves. |
   170→| Direct API calls to Medusa or ERPNext | All backend calls go through `services/`. Pages never import `lib/medusa.ts` directly. |
   171→| Complex components | Pages compose feature modules. If a page file exceeds 80 lines, extract a feature component. |
   172→| State management | State lives in `features/` or `hooks/`. Pages read state, they do not define it. |
   173→
   174→### Full Structure
   175→
   176→```
   177→app/
   178→├── (shop)/                           # Public storefront route group
   179→│   ├── layout.tsx                    # Shop layout: header + footer + nav

   255→```
   256→
   257→### Best Practices
   258→
   259→| Practice | Detail |
   260→|---|---|
   261→| **One concern per route** | `page.tsx` files compose features. They import from `features/`, pass props, and handle metadata. No business logic. |
   262→| **Server Components by default** | Every page is a Server Component unless it needs interactivity. Mark `'use client'` only when required. |
   263→| **Loading states for every route** | Every route that fetches data has a `loading.tsx` sibling with a skeleton matching the final layout shape. No blank screens. |
   264→| **RTL in the root layout** | `layout.tsx` reads locale from the session and sets `<html dir="rtl" lang="ar">` or `<html dir="ltr" lang="en">`. This is set once, at the root, never overridden. |
   265→| **API routes are thin** | An API route validates input (Zod), calls a service function, and returns a response. Maximum 30 lines. If it grows, move logic to `services/`. |

   269→## 5. frontend/src/components/ — UI Components
   270→
   271→### Purpose
   272→
   273→The `components/` directory holds the **design system**: reusable, domain-agnostic UI primitives. These components know nothing about products, carts, or orders. They know about buttons, cards, inputs, and modals.
   274→
   275→### What Belongs Here
   276→
   277→- shadcn/ui components (installed and customized)
   278→- Custom wrappers around shadcn/ui with GGH-specific styling
   279→- Layout primitives (Container, Section, Grid)
   280→- Shared structural components (Header, Footer, Navigation)
   281→- Accessible form controls with RTL support built in
   282→
   283→### What Should NOT Be Here
   284→
   285→| Anti-pattern | Where It Belongs Instead |
   286→|---|---|
   287→| Components that display product data | `features/product/components/` |
   288→| Components that read from cart state | `features/cart/components/` |
   289→| Components that call APIs | `features/*/components/` or `services/` |
   290→| Page-specific layouts | `app/(group)/layout.tsx` |
   291→| One-off components used in a single feature | That feature's `components/` folder |
   292→
   293→### Structure
   294→
   295→```
   296→components/
   297→├── ui/                               # shadcn/ui primitives (auto-generated)
   298→│   ├── button.tsx
   299→│   ├── card.tsx
   300→│   ├── input.tsx
   301→│   ├── select.tsx
   302→│   ├── sheet.tsx                     # Slide-out panel (used for cart)
   303→│   ├── dialog.tsx
   304→│   ├── badge.tsx
   305→│   ├── separator.tsx
   306→│   ├── skeleton.tsx
   307→│   ├── toast.tsx
   308→│   ├── dropdown-menu.tsx
   309→│   ├── accordion.tsx
   310→│   ├── tabs.tsx
   311→│   ├── checkbox.tsx
   312→│   ├── radio-group.tsx
   313→│   ├── stepper.tsx                   # Custom: quantity stepper with +/−
   314→│   └── countdown.tsx                 # Custom: deal countdown timer
   315→│
   316→├── layout/                           # Structural layout components
   317→│   ├── container.tsx                 # Max-width centered wrapper
   318→│   ├── section.tsx                   # Page section with heading + optional "View All"
   319→│   ├── grid.tsx                      # Responsive product grid
   320→│   └── scroll-row.tsx                # Horizontal scroll container (categories, deals)
   321→│

   324→│   ├── footer.tsx                    # Footer: links, contact, social
   325→│   ├── mobile-nav.tsx                # Mobile bottom navigation bar
   326→│   ├── category-nav.tsx              # Category horizontal scroll nav
   327→│   └── breadcrumbs.tsx               # RTL-aware breadcrumb trail
   328→│

   330→│   ├── loading-spinner.tsx           # Inline loading indicator
   331→│   ├── empty-state.tsx               # "No items yet" with illustration + CTA
   332→│   ├── error-state.tsx               # "Something went wrong" with retry button
   333→│   ├── savings-badge.tsx             # "Save X EGP" / "وفّر X ج.م" badge
   334→│   └── price-display.tsx             # Today/yesterday price with savings calc
   335→│
   336→└── index.ts                          # Barrel export for all components
   337→```
   338→
   339→### Best Practices
   340→
   341→| Practice | Detail |
   342→|---|---|
   343→| **No business logic** | A `ProductCard` does not live here. A `Card` does. The difference: `Card` accepts `title`, `image`, `price` as props. It does not know they come from a product. |
   344→| **RTL built in** | Every component that renders directionally (stepper, price display, badge with icon) uses `rtl:` Tailwind variants. Never `ml-` or `mr-` — use `ms-` and `me-` (logical properties). |
   345→| **Touch targets** | Every interactive element meets the 48px minimum. If a shadcn/ui component falls short, wrap it in a `TouchTarget` component that adds padding. |
   346→| **Controlled + accessible** | Every form control accepts `aria-label` in both languages. Every button has a visible text label (no icon-only buttons except in the header, where `aria-label` is mandatory). |

   350→## 6. frontend/src/features/ — Domain Modules
   351→
   352→### Purpose
   353→
   354→The `features/` directory is the heart of the application. Each subfolder is a **domain module** that groups everything related to one business capability: components, hooks, types, utils, and constants. Features are self-contained and do not import from each other.
   355→
   356→### What Belongs Here
   357→
   358→Everything specific to a business domain: product display, cart management, checkout flow, order history, deal management, customer account, search, and wholesale operations.
   359→
   360→### What Should NOT Be Here
   361→
   362→| Anti-pattern | Where It Belongs Instead |
   363→|---|---|
   364→| Generic UI components (buttons, cards) | `components/ui/` |
   365→| Shared hooks used across features | `hooks/` |
   366→| API client configuration | `lib/` |
   367→| Pure utility functions (formatEGP, formatDate) | `utils/` |
   368→| Cross-cutting types (User, Locale) | `types/` |

   492→
   493→| Rule | Enforcement |
   494→|---|---|
   495→| **No cross-feature imports** | `features/product/` never imports from `features/cart/`. If product-card needs to add to cart, it calls `useCart().addItem()` from the shared `hooks/` layer, or receives an `onAddToCart` callback. |
   496→| **Every feature is a leaf** | Features are consumed by `app/` pages. Nothing depends on features except pages and layouts. |
   497→| **Shared state via hooks** | Cart state lives in `features/cart/store/`. Any feature that needs cart data imports `use-cart.ts` from `features/cart/hooks/`. This is the one exception to cross-feature imports: cart hooks are the public API of the cart feature. |
   498→| **Feature types are local** | `features/product/types/product.types.ts` defines the Product type used within that feature. If the type is needed globally, it moves to `types/`. |

   502→## 7. frontend/src/hooks/ — Custom React Hooks
   503→
   504→### Purpose
   505→
   506→Shared React hooks that are **not specific to one feature**. Feature-specific hooks live inside their `features/*/hooks/` directory. Only hooks used by two or more features belong here.
   507→
   508→### What Belongs Here
   509→
   510→| Hook | Purpose |
   511→|---|---|
   512→| `use-locale.ts` | Current locale, direction (RTL/LTR), and locale toggle function |
   513→| `use-debounce.ts` | Debounce a value (search input, resize events) |
   514→| `use-media-query.ts` | Responsive breakpoint detection |
   515→| `use-intersection.ts` | Intersection Observer for lazy loading and infinite scroll |
   516→| `use-keyboard.ts` | Keyboard shortcut handler for accessibility |
   517→| `use-click-outside.ts` | Detect clicks outside an element (close dropdowns) |
   518→| `use-local-storage.ts` | Persist state in localStorage with type safety |
   519→| `use-async.ts` | Generic async state handler (loading, error, data) |
   520→
   521→### What Should NOT Be Here
   522→
   523→| Anti-pattern | Where It Belongs |
   524→|---|---|
   525→| `use-cart.ts` | `features/cart/hooks/` (domain-specific) |
   526→| `use-products.ts` | `features/product/hooks/` (domain-specific) |
   527→| `useSWR` or `useQuery` wrappers | `lib/` (configuration, not hooks) |
   528→| Non-React functions | `utils/` |
   529→
   530→### Example File
   531→
   532→```typescript
   533→// hooks/use-locale.ts
   534→import { useSearchParams } from 'next/navigation';
   535→import { useCallback } from 'react';
   536→
   537→type Locale = 'en' | 'ar';
   538→type Direction = 'ltr' | 'rtl';
   539→
   540→export function useLocale() {
   541→  const searchParams = useSearchParams();
   542→  const locale: Locale = (searchParams.get('lang') as Locale) ?? 'ar';
   543→  const direction: Direction = locale === 'ar' ? 'rtl' : 'ltr';
   544→
   545→  const toggleLocale = useCallback(() => {
   546→    const newLocale = locale === 'ar' ? 'en' : 'ar';
   547→    // Set cookie and trigger navigation
   548→    document.cookie = `ggh_locale=${newLocale};path=/;max-age=${60 * 60 * 24 * 365}`;
   549→    window.location.reload();
   550→  }, [locale]);
   551→
   552→  return { locale, direction, toggleLocale } as const;
   553→}
   554→```

   624→## 9. frontend/src/services/ — API Service Layer
   625→
   626→### Purpose
   627→
   628→The `services/` directory contains **one file per backend resource**. Each file exports async functions that call the BFF API routes (`/api/*`). These functions are the only way frontend code interacts with the backend. No component or hook ever calls `fetch('/api/...')` directly.
   629→
   630→### What Belongs Here
   631→
   632→| File | Functions |
   633→|---|---|
   634→| `product.service.ts` | `getProducts()`, `getProductByHandle()`, `getProductsByCategory()` |
   635→| `category.service.ts` | `getCategories()`, `getCategoryBySlug()` |
   636→| `cart.service.ts` | `getCart()`, `addToCart()`, `updateCartItem()`, `removeCartItem()` |
   637→| `order.service.ts` | `getOrders()`, `getOrderById()`, `reorder()` |
   638→| `auth.service.ts` | `login()`, `register()`, `logout()`, `getSession()` |
   639→| `checkout.service.ts` | `initiateCheckout()`, `completeCheckout()` |
   640→| `search.service.ts` | `searchProducts()` |
   641→| `deal.service.ts` | `getActiveDeals()` |
   642→
   643→### What Should NOT Be Here
   644→
   645→| Anti-pattern | Where It Belongs |
   646→|---|---|
   647→| Direct Medusa or ERPNext API calls | `lib/` clients are used in API routes, not in services |
   648→| State management | `features/*/store/` or `hooks/` |
   649→| Component rendering logic | `features/*/components/` |
   650→| Business logic (price calculation) | `features/*/utils/` |
   651→
   652→### Example File
   653→
   654→```typescript
   655→// services/product.service.ts
   656→import type { Product, ProductListResponse } from '@/types/product';
   657→
   658→const API_BASE = '/api/products';
   659→
   660→export async function getProducts(params?: {
   661→  categoryId?: string;
   662→  query?: string;
   663→  page?: number;
   664→  limit?: number;
   665→}): Promise<ProductListResponse> {
   666→  const searchParams = new URLSearchParams();
   667→  if (params?.categoryId) searchParams.set('categoryId', params.categoryId);
   668→  if (params?.query) searchParams.set('q', params.query);
   669→  if (params?.page) searchParams.set('page', String(params.page));
   670→  if (params?.limit) searchParams.set('limit', String(params.limit));
   671→
   672→  const res = await fetch(`${API_BASE}?${searchParams.toString()}`);
   673→  if (!res.ok) throw new Error('Failed to fetch products');
   674→  return res.json();
   675→}
   676→
   677→export async function getProductByHandle(handle: string): Promise<Product> {
   678→  const res = await fetch(`${API_BASE}/${handle}`);
   679→  if (!res.ok) throw new Error('Failed to fetch product');
   680→  return res.json();
   681→}
   682→```
   683→
   684→### Best Practices
   685→
   686→| Practice | Detail |
   687→|---|---|
   688→| **One file per resource** | `product.service.ts` handles all product API calls. Do not split into `product-get.service.ts` and `product-create.service.ts`. |
   689→| **Type every function** | Every service function has a typed return value using types from `types/`. No `any`. |
   690→| **Handle errors consistently** | Every function throws on non-2xx. The calling hook or component catches and displays the error. |
   691→| **No caching logic** | Services are pure data fetchers. Caching is handled by TanStack Query configuration in the calling hook. |

   991→### Best Practices
   992→
   993→| Practice | Detail |
   994→|---|---|
   995→| **Arabic file is not a translation of English** | The Arabic file is written by an Arabic-speaking team member. Machine translation is forbidden for UI strings. |
   996→| **Every key has both languages** | CI checks that `en.json` and `ar.json` have identical key sets. Missing keys fail the build. |
   997→| **Use ICU message format** | For interpolation (`{count}`, `{amount}`), plural rules (`{count, plural, one{item} other{items}}`), and RTL-aware formatting. |
   998→| **No hardcoded strings in components** | Every user-facing string comes from the translation file. Developer-facing strings (console.error, log messages) stay in English. |

  1058→## 15. frontend/src/utils/ — Pure Utility Functions
  1059→
  1060→### Purpose
  1061→
  1062→Pure functions with **zero side effects** and **zero external dependencies** (no React, no API calls, no DOM access). These are the most reusable and testable functions in the codebase.
  1063→
  1064→### What Belongs Here
  1065→
  1066→| File | Functions |
  1067→|---|---|
  1068→| `currency.ts` | `formatEGP()`, `parsePiastres()`, `calculateSavings()` |
  1069→| `date.ts` | `formatDate()`, `formatRelativeTime()`, `getDeliveryETA()` |
  1070→| `string.ts` | `truncate()`, `slugify()`, `normalizeArabic()` |
  1071→| `number.ts` | `clamp()`, `roundToNearest()`, `calculatePriceTier()` |
  1072→| `validation.ts` | `isValidEgyptianPhone()`, `isValidEmail()`, `meetsMOQ()` |
  1073→| `object.ts` | `pick()`, `omit()`, `deepMerge()` |
  1074→| `array.ts` | `groupBy()`, `uniqueBy()`, `paginate()` |
  1075→
  1076→### What Should NOT Be Here
  1077→
  1078→| Anti-pattern | Where It Belongs |
  1079→|---|---|
  1080→| React hooks | `hooks/` |
  1081→| API calls | `services/` |
  1082→| Functions that read from `localStorage` or `document` | `hooks/` (side effects belong in hooks, not utils) |
  1083→| Functions that import from `lib/`, `services/`, or `features/` | Utils are leaf nodes. They import nothing internal. |
  1084→| Business logic with domain knowledge | `features/*/utils/` |
  1085→
  1086→### Example File
  1087→
  1088→```typescript
  1089→// utils/currency.ts
  1090→
  1091→/** Format piastres as Egyptian Pounds string */
  1092→export function formatEGP(piastres: number, locale: 'en' | 'ar' = 'ar'): string {
  1093→  const pounds = piastres / 100;
  1094→  if (locale === 'ar') {
  1095→    return `${pounds.toFixed(2)} ج.م`;
  1096→  }
  1097→  return `${pounds.toFixed(2)} EGP`;
  1098→}
  1099→
  1100→/** Calculate savings between two prices (in piastres) */
  1101→export function calculateSavings(todayPrice: number, yesterdayPrice: number): {
  1102→  savingsAmount: number;
  1103→  savingsPercent: number;
  1104→} {
  1105→  const savingsAmount = yesterdayPrice - todayPrice;
  1106→  const savingsPercent = Math.round((savingsAmount / yesterdayPrice) * 100);
  1107→  return { savingsAmount, savingsPercent };
  1108→}
  1109→
  1110→/** Check if a quantity meets the minimum order quantity */
  1111→export function meetsMOQ(quantity: number, moq: number): boolean {
  1112→  return quantity >= moq;
  1113→}
  1114→```

  1336→
  1337→| Practice | Detail |
  1338→|---|---|
  1339→| **Multi-stage builds** | Separate `deps` stage (install everything) from `build` stage (compile TypeScript) from `production` stage (copy only what's needed). Keeps images small. |
  1340→| **Non-root user** | Production stage runs as `nextjs` or `node` user, never `root`. |
  1341→| **Health checks** | Every Dockerfile includes a `HEALTHCHECK` instruction that hits the service's `/health` endpoint. |
  1342→| **Layer caching** | Copy `package.json` and lock file first, install deps, then copy source code. Code changes don't invalidate the deps layer. |

  1446→| Docker files | `Dockerfile` (capital D) | `docker/nextjs/Dockerfile` |
  1447→| Environment templates | `.env.example` | `frontend/.env.example` |
  1448→
  1449→### Directory Naming
  1450→
  1451→| Pattern | Convention | Example |
  1452→|---|---|---|
  1453→| Route groups | `(kebab-case)` — parentheses in Next.js | `(shop)/`, `(auth)/` |
  1454→| Dynamic routes | `[kebab-case]` — brackets in Next.js | `[slug]/`, `[handle]/` |
  1455→| Feature directories | `kebab-case` | `features/product/`, `features/order-template/` |
  1456→| Component directories | `kebab-case` | `components/navigation/` |

  1460→## 22. Import Rules & Boundaries
  1461→
  1462→### Allowed Import Directions
  1463→
  1464→```
  1465→app/         ← imports from: features/, components/, providers/, lib/, types/
  1466→features/    ← imports from: components/, hooks/, services/, lib/, utils/, types/
  1467→components/  ← imports from: lib/, utils/, types/
  1468→hooks/       ← imports from: services/, lib/, utils/, types/
  1469→services/    ← imports from: lib/, types/, utils/
  1470→lib/         ← imports from: types/, utils/
  1471→providers/   ← imports from: hooks/, lib/, types/
  1472→utils/       ← imports from: NOTHING (pure functions only)
  1473→types/       ← imports from: utils/ (rarely)
  1474→locales/     ← no imports (data files only)
  1475→styles/      ← no imports (CSS only)
  1476→```
  1477→
  1478→### Forbidden Imports
  1479→
  1480→| From | To | Why |
  1481→|---|---|---|
  1482→| `components/` | `features/` | UI primitives must not depend on domain logic |
  1483→| `utils/` | `lib/`, `services/`, `hooks/` | Utils are leaf nodes. Zero internal dependencies. |
  1484→| `services/` | `features/` | Services are data fetchers, not domain-aware |
  1485→| `features/product/` | `features/cart/` | No cross-feature imports (except cart hooks — see below) |
  1486→| Any file | `node_modules` direct import without barrel | All third-party imports should go through a named import |
  1487→
  1488→### The Cart Exception
  1489→
  1490→The cart is unique because multiple features need to add items to it (product card, deal card, reorder button). The cart feature exposes its hooks as a **public API**:
  1491→
  1492→```typescript
  1493→// features/cart/hooks/use-cart.ts — PUBLIC API
  1494→export function useCart() {
  1495→  // ... implementation
  1496→  return { cart, addItem, updateItem, removeItem, summary } as const;
  1497→}
  1498→```
  1499→
  1500→Other features may import `useCart` from `features/cart/hooks/use-cart`. This is the **only** permitted cross-feature import. All other cross-feature communication happens through:
  1501→- **Props** (parent page passes callbacks)
  1502→- **Events** (BullMQ on the backend)
  1503→- **Shared state** (Zustand store in `features/cart/store/`)

  1507→## 23. Adding a New Feature — Checklist
  1508→
  1509→When a developer adds a new feature (e.g., "Wishlist"), follow this checklist:
  1510→
  1511→```
  1512→1. Create the feature directory
  1513→   mkdir -p frontend/src/features/wishlist/{components,hooks,types,utils}
  1514→
  1515→2. Define types
  1516→   Create frontend/src/features/wishlist/types/wishlist.types.ts
  1517→   If types are shared across features, move to frontend/src/types/
  1518→
  1519→3. Create service functions
  1520→   Add wishlist functions to frontend/src/services/wishlist.service.ts
  1521→
  1522→4. Create hooks
  1523→   Create frontend/src/features/wishlist/hooks/use-wishlist.ts
  1524→
  1525→5. Create components
  1526→   Create frontend/src/features/wishlist/components/wishlist-card.tsx
  1527→   Create frontend/src/features/wishlist/components/wishlist-page.tsx
  1528→
  1529→6. Add translations
  1530→   Add "wishlist" namespace to frontend/src/locales/en.json and ar.json
  1531→
  1532→7. Add API route (if needed)
  1533→   Create frontend/src/app/api/wishlist/route.ts
  1534→
  1535→8. Add page route
  1536→   Create frontend/src/app/(account)/wishlist/page.tsx
  1537→
  1538→9. Verify import boundaries
  1539→   - No import from other features (except useCart if adding to cart)
  1540→   - No business logic in page.tsx
  1541→   - All API calls go through service functions
  1542→
  1543→10. Verify accessibility
  1544→    - 48px touch targets on all interactive elements
  1545→    - Arabic labels on all elements
  1546→    - RTL layout tested with dir="rtl"
  1547→    - Screen reader announces state changes (added/removed from wishlist)
  1548→```

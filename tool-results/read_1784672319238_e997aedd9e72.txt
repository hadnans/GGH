    37→## 1. Implementation Philosophy
    38→
    39→| Principle | Rule | Why |
    40→|---|---|---|
    41→| **Sequential and testable** | Each milestone produces a runnable system. No milestone leaves the backend in a broken state. | Jules needs a green build after every milestone. Partial progress must still work. |
    42→| **No code in this document** | This document specifies WHAT to build, not HOW to write it. No TypeScript, no SQL, no YAML. | Implementation is the engineer's job. This is the specification. |
    43→| **Dependencies are explicit** | Every milestone lists what must be complete before it starts. No implicit assumptions. | Jules must never encounter "oh, you also need X" mid-milestone. |
    44→| **Acceptance criteria are binary** | Every feature has pass/fail acceptance criteria. No "mostly works" or "looks good." | A criterion is either met or not met. No judgment calls. |
    45→| **Elder-friendly is non-negotiable** | Every API response, error message, and timeout budget accounts for slow connections and low-tech users. | If Om Ibrahim cannot complete the flow, the feature is not done. |
    46→| **Money is always integer piastres** | No floating-point monetary values anywhere. EGP 110.50 is stored and transmitted as 11050. | Floating-point errors in money are unacceptable. This is non-negotiable. |
    47→| **Bilingual is always both** | Every user-facing string returned by the API includes both `_en` and `_ar` fields. No "default to English." | Arabic is the primary language. Missing Arabic is a bug, not a feature gap. |
    48→| **ERPNext is the source of truth** | Inventory and base pricing come from ERP. GGH is an operational mirror. | When in doubt, ERP wins for data. GGH wins for customer-facing experience. |

    52→## 2. Prerequisite Conventions
    53→
    54→Before any milestone, the following conventions are assumed throughout:
    55→
    56→### 2.1 Repository Structure
    57→
    58→```
    59→backend/
    60→├── src/
    61→│   ├── api/                  # Medusa API routes
    62→│   │   ├── admin/            # Admin-facing routes
    63→│   │   └── store/            # Customer-facing routes (BFF calls these)
    64→│   ├── modules/              # Custom Medusa modules
    65→│   │   ├── ggh-delivery/     # Delivery zones & slots
    66→│   │   ├── ggh-pricing/      # Price tiers & bulk-break
    67→│   │   ├── ggh-wholesale/    # Wholesale accounts & verification
    68→│   │   ├── ggh-otp/          # OTP authentication module
    69→│   │   └── ggh-notification/ # Notification preferences & dispatch
    70→│   ├── jobs/                 # BullMQ job processors
    71→│   │   ├── erp/              # ERP sync jobs
    72→│   │   ├── notifications/    # Notification dispatch jobs
    73→│   │   ├── orders/           # Order processing jobs
    74→│   │   └── cache/            # Cache invalidation jobs
    75→│   ├── subscribers/          # Medusa event subscribers
    76→│   ├── loaders/              # Custom Medusa loaders
    77→│   └── utils/                # Shared utilities
    78→├── migrations/               # GGH schema migrations (in ggh/ subdirectory)
    79→├── prisma/                   # Prisma schema for ggh database access
    80→├── medusa-config.ts          # Medusa configuration
    81→├── .env.example              # Environment variable template
    82→└── Dockerfile
    83→```
    84→
    85→### 2.2 Naming Conventions
    86→
    87→| Item | Convention | Example |
    88→|---|---|---|
    89→| Medusa module directory | `ggh-{domain}` | `ggh-delivery`, `ggh-pricing` |
    90→| BullMQ queue name | `erp.sync.{entity}` or `erp.push.{entity}` | `erp.sync.inventory`, `erp.push.order` |
    91→| BullMQ job file | `{queueName}.processor.ts` | `erp-sync-inventory.processor.ts` |
    92→| Database migration | `YYYYMMDDHHMMSS_description.ts` | `20250116140000_create_delivery_zones.ts` |
    93→| API route file | `index.ts` in route directory | `api/store/delivery-zones/index.ts` |
    94→| Subscriber file | `{event}.{handler}.subscriber.ts` | `order.placed.erp-push.subscriber.ts` |
    95→| Environment variable | `GGH_{DOMAIN}_{VARIABLE}` | `GGH_ERP_API_KEY`, `GGH_SMS_PROVIDER` |
    96→| Redis key prefix | `cache:`, `sess:`, `otp:`, `bull:` | `cache:products:category:rice` |
    97→
    98→### 2.3 Database Schema Separation
    99→
   100→| Schema | Owner | Rule |
   101→|---|---|---|
   102→| `public` | Medusa | GGH never modifies Medusa tables directly. Access through Medusa API only. |
   103→| `ggh` | GGH | GGH manages all tables in this schema via Prisma and custom migrations. |
   104→
   105→### 2.4 Cross-Schema References
   106→
   107→GGH tables reference Medusa entities by storing Medusa IDs as `TEXT` columns. These are **not** foreign keys — they are application-level references validated in code, not by the database.
   108→
   109→| GGH Table | Medusa Entity | Column | Type |
   110→|---|---|---|---|
   111→| `ggh.price_tier` | `public.product_variant` | `variant_id` | TEXT |
   112→| `ggh.order_template` | `public.customer` | `customer_id` | TEXT |
   113→| `ggh.wholesale_account` | `public.customer` | `customer_id` | TEXT |
   114→| `ggh.delivery_proof` | `public.order` | `order_id` | TEXT |
   115→| `ggh.notification_preference` | `public.customer` | `customer_id` | TEXT |
   116→| `ggh.erp_sync_log` | various | `entity_id` | TEXT |
   117→| `ggh.order_status_log` | `public.order` | `order_id` | TEXT |
